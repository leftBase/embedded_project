스레드는 어떤 단위인가? (설계 + 관리)

권장 스레드 분할:
Input Thread: 키 입력을 읽어 입력 큐/플래그에 저장(비차단). 하나로 충분하거나 각 컨트롤러별로 하나.
Game Logic Thread(또는 타이머): 고정 틱(예: 30FPS)으로 상태 갱신(스폰, 이동, 충돌).
Render Thread: ncurses 호출만 담당(유일하게 ncurses 접근).
옵션: Sound/IO Thread, Network Thread 등.
관리 방법:
스레드 핸들(pthread_t)을 저장하고, 종료 시 pthread_join으로 정리.
상태 모니터링: GameLog나 atomic 변수(예: running, tick_count)를 갱신하여 로그로 확인.
디버깅: 각 스레드의 주기마다 간단한 로그(레벨=디버그) 기록.
## 뮤텍스 사용법(주의 포인트)

간단 규칙:
공유 변수는 항상 뮤텍스로 보호하거나 원자 타입 사용.
락은 가능한 좁은 범위에서만 잡기.
여러 락을 잡을 때는 항상 정해진 순서로만 잡아 데드락 방지.
예: 점수 증가 안전하게 하기

    gs.players[player_id].score += delta;

}
입력 최적화: 단순 플래그(예: atomic_int last_input)는 뮤텍스 없이 stdatomic.h로 처리 가능.

## 장애물 모델링: 정수 위치 vs lifetime(타이머) — 권장안

요구: 장애물은 레인(1/2/3)과 플레이어와의 거리만 필요. 나타나면 이동 → 사라짐.
권장: 틱 기반의 정수 위치(또는 고정소수점) + 속도(틱당 이동량).
이유: 구현 단순, 충돌 판정(같은 레인 && pos 비교)이 직관적.
구조체 예:
fractional speed 필요하면 fixed-point(예: pos in centi-units, speed in centi-units/tick) 사용. float 써도 무방.
lifetime 방식(생성시간 + TTL) 장점: 스폰/사라짐 시점 계산 편리. 단, 충돌 판정은 position을 계산해야 하므로 결국 위치 계산이 필요.
결론: 틱 기반 위치 + 속도(정수/고정소수점)가 가장 단순하고 성능/정확성 균형이 좋음.
충돌 판정(간단 룰)

조건: obs.lane == player.lane && abs(obs.pos - PLAYER_POS) <= THRESHOLD
THRESHOLD는 플레이어와 장애물의 논리적 크기(보통 0 또는 1).
한 틱 내에 속도가 커서 관통 가능하면 swept 테스트(이전 pos → 현재 pos 구간과 player pos 비교) 필요.
speed는 어디에 둬야 하는가?

플레이어 속도: 플레이어별 고유 특성 → Player.speed.
장애물 속도: 장애물별 고유 특성 → Obstacle.speed.
GameState에 global_speed_multiplier 같은 전역 난이도 계수는 둘 수 있음.
렌더가 별도 GameState 복사본을 갖는 건 권장하지 않음(복사본 동기화 복잡). 렌더는 읽기 전용으로 공유 GameState를 복사해서 사용.
## 렌더에서의 처리(권장)

render.c는 오직 화면 표현(문자열, 색)만 담당.
렌더가 직접 게임 로직(생성/충돌)을 하지 않음.
성능 개선: 렌더는 gs를 잠깐 복사한 뒤 잠금 해제하고 그 복사본으로 지속 렌더링.
요약(짧게):

플레이어와 게임상태는 분리: Player와 GameState.
뮤텍스는 GameState 전체(또는 per-player + obstacle mutex) 보호. 락 시간 최소화.
장애물은 틱 기반 pos + speed(정수 또는 fixed-point)로 처리하길 권장.
렌더는 읽기 전용, ncurses 호출은 렌더 스레드만 수행.
speed는 플레이어 속성은 Player에, 장애물 속성은 Obstacle에.

- `GameState` : 전역 상태(플레이어 배열, 장애물 리스트, 게임 상태, 전역 난이도)

**2) ncurses와 테라텀(터미널) 호환성**
- 조건:
  - 임베디드 보드가 Linux이고 ncurses 라이브러리(.so/.a)가 설치되어 있어야 함.
  - Tera Term은 ANSI escape 시퀀스를 지원하므로, 임베디드에서 stdout으로 ncurses 출력(터미널 이스케이프)을 보내면 PC 쪽 Tera Term이 화면을 잘 렌더링함.
  - 연결 방식:
    - SSH: ncurses 완전 호환.
    - Serial(시리얼 콘솔): 대부분의 경우 호환. 단, 임베디드에서 콘솔로 로그인된 세션(또는 프로그램의 stdout/stderr/stdin)이 시리얼 포트에 바인딩되어야 함.
- 요약: 임베디드 Linux + ncurses 설치 + Tera Term(또는 다른 터미널) → 가능. 만약 ncurses가 없다면 터미널 제어를 직접 쓰거나 간단한 텍스트 UI로 구현.

**3) 스레드는 무엇인가? (중괄호 단위 아님)**
- 정의: 스레드는 OS가 스케줄링하는 실행 단위(프로세스의 경량 실행 단위). 각 스레드는 독립된 스택을 가지며 같은 주소 공간(전역 변수, 힙)을 공유.
- 코드 블록(중괄호)은 단지 범위(scope)일 뿐 스레드와 무관. 스레드는 함수(또는 함수 내부의 코드)를 실행할 때 생성됨.
- 실무 포인트:
  - 스레드 생성: `pthread_create` → 반환된 `pthread_t`로 관리(종료 대기 `pthread_join` 또는 `pthread_detach`).
  - 스레드 종료: 함수 리턴 또는 `pthread_exit`.
  - 스택 크기/우선순위/스케줄링 정책은 필요하면 설정 가능(권한 필요).

**4) 뮤텍스 실전 예시 (핵심만)**
- 원칙: 공유 상태(예: `GameState`)는 뮤텍스로 보호. 락은 가능한 짧게 유지. 렌더는 읽기용 스냅샷 복사 후 락 해제.
- 간단 구현 골격:

```c
#include <pthread.h>
#include <stdatomic.h>
#include <stdbool.h>

typedef struct { int lane, speed, score, life; } Player;
typedef struct { int lane, pos, speed; bool active; } Obstacle;
typedef struct {
    Player players[2];
    Obstacle obstacles[64];
    int obstacle_count;
    int state; // RUNNING/PAUSED/OVER
    int global_speed_mul;
} GameState;

static GameState gs;
static pthread_mutex_t gs_lock = PTHREAD_MUTEX_INITIALIZER;
static atomic_bool running = true;

/* Input thread: 입력을 받아 최소한의 변경만 수행 */
void *input_thread(void *arg) {
    while (atomic_load(&running)) {
        int ch = getch(); // nodelay 모드에서 사용 가능 (ncurses)
        if (ch != ERR) {
            pthread_mutex_lock(&gs_lock);
            // 예: 플레이어 0 이동
            if (ch == 'a' && gs.players[0].lane > 0) gs.players[0].lane--;
            if (ch == 'd' && gs.players[0].lane < 2) gs.players[0].lane++;
            pthread_mutex_unlock(&gs_lock);
        }
        usleep(10000);
    }
    return NULL;
}

/* Game logic thread: 모든 상태 갱신(생성/이동/충돌) 수행 */
void *game_thread(void *arg) {
    while (atomic_load(&running)) {
        pthread_mutex_lock(&gs_lock);
        // 장애물 이동, 충돌 검사, 스폰 등
        for (int i=0;i<gs.obstacle_count;i++){
            if (!gs.obstacles[i].active) continue;
            gs.obstacles[i].pos += gs.obstacles[i].speed;
            for (int p=0;p<2;p++){
                if (gs.players[p].lane == gs.obstacles[i].lane &&
                    gs.obstacles[i].pos >= /*player_pos*/10) {
                    gs.players[p].life--;
                    gs.obstacles[i].active = false;
                }
            }
        }
        pthread_mutex_unlock(&gs_lock);
        usleep(33333); // 틱
    }
    return NULL;
}

/* Render thread: 락으로 스냅샷 복사 후 ncurses로 렌더 */
void *render_thread(void *arg) {
    GameState snap;
    while (atomic_load(&running)) {
        pthread_mutex_lock(&gs_lock);
        snap = gs; // 얕은 복사 (크면 필요한 부분만 복사)
        pthread_mutex_unlock(&gs_lock);
        // ncurses로 snap 그리기 (render.c)
        refresh();
        usleep(16666);
    }
    return NULL;
}
```

- 뮤텍스 사용 팁:
  - 단일 전역 락이 가장 단순. 성능 문제 시 finer-grained locks(예: per-player, obstacle list) 고려.
  - `pthread_mutex_trylock`으로 블로킹 방지 로직 구현 가능.
  - 상태 플래그(예: `running`)는 `atomic_bool`로 처리하면 간단.

**5) 입력(시리얼) 실무 처리**
- 상황 A: 프로그램이 임베디드에서 실행되고 Tera Term(PC)이 시리얼로 연결되어 있다.
  - 표준 입력(stdin)이 시리얼 콘솔에 바인딩되어 있다면 ncurses `getch()`가 작동.
  - 또는 직접 `/dev/ttySx` 를 `open()` 하고 `termios`로 라인모드/비라인모드 설정 후 `read()` 하여 처리.
- 상황 B: 테라텀 → 임베디드가 원격입력 전송(터미널 에뮬레이션)이라면 ncurses는 정상 작동.
- 권장: ncurses + `nodelay(stdscr, TRUE)` + `keypad(stdscr, TRUE)` 를 사용하고, 입력은 전용 input thread가 처리. 시리얼 특성(라인 엔딩 등)에 유의.

**6) 장애물 모델: 정수 pos + speed(틱당 증가) 권장**
- 구조체 가벼움 예:
```c
typedef struct {
  int lane;    // 0..2
  int pos;     // 정수 위치(틱 단위)
  int speed;   // units per tick (정수, 필요시 fixed-point)
  bool active;
} Obstacle;
```
- 이유: 단순하고 충돌 계산이 빠름. lifetime(TTL) 방식은 스폰/사라짐만 필요하면 가능하지만 충돌에서 위치 계산이 결국 필요함.
- 관통(고속) 문제: 한 틱에 지나치게 멀리 이동하면 이전 위치→현재 위치 구간을 검사하는 swept 충돌 검사 필요.

**7) `speed`는 어디에 둬야 하나?**
- 플레이어 고유 속도 → `Player.speed`.
- 장애물 고유 속도 → `Obstacle.speed`.
- 전역 난이도/속도 보정 → `GameState.global_speed_mul` 또는 타이머에 반영.
- Render는 속성을 갖지 않음(렌더는 표현만). 렌더가 별도 gamestate를 갖는 건 동기화 부담 증가.

**8) 스레드 관리/모니터링(실무)**

- 정상 종료: `atomic_bool running = false; pthread_join(...)`
- 로그/헬스: 각 스레드에 주기적 heartbeat 로그(또는 tick_count)를 남겨 모니터링.
